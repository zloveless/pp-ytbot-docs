1. Overview

ytbot is a YouTube chat integration bot for 7 Days to Die (7DTD). Viewers spend accumulated points to trigger in-game events — debuffs, heals, zombie spawns — by typing commands in the YouTube live chat. This documentation site will serve as the viewer-facing reference for the points system, available commands, costs, and behavior rules.

The authoritative source for the initial content is YTbot Commands-VanillaVersion-EarlyGame.rtf. The filename encodes two scope signals that directly inform the site architecture:

  • VanillaVersion — a modded variant with a different command set may follow.
  • EarlyGame — costs and command availability may change for mid/late game.
Design constraint: The content model must be version- and stage-aware from day one. Hardcoding a single table of commands/costs will require a rewrite the moment a second RTF appears. See §4 (Content Model) for the extensibility approach.

Command Data Model

Each command entry from the RTF carries these fields. This becomes the schema for the data files the SSG consumes:

FieldTypeNotes
keywordstringChat trigger, e.g. burn
categoryenumnegative · positive · spawn
effectstringHuman-readable description of what happens in-game
costnumberPoint cost to activate
targetenumself · player · all
superChatMinnumber?Minimum Super Chat USD equivalent, if applicable
versionstringe.g. vanilla, modded
stagestringe.g. early, mid, late
notesstring?Edge-case behavior, duration, etc.

For spawn commands, two additional fields apply:

FieldTypeNotes

extraZombieThresholdnumberAdditional viewers required per +1 spawn

3. Tooling Recommendation

Astro

  • Zero JS by default; ships pure HTML — ideal for a reference doc
  • First-class content collections with typed frontmatter (.yml or .json data files)
  • Starter templates install in one command; Yarn 4 compatible
  • Markdown + MDX pages; no framework lock-in
  • Built-in astro build → static output; trivially deployable

Eleventy (11ty)

  • Extremely lightweight; no JS framework
  • Flexible templating (Nunjucks, Liquid, Markdown)
  • Data cascade well-suited to versioned JSON command sets
  • Less opinionated — more manual wiring required
  • Good choice if Astro feels over-engineered for a small site

Recommendation: Astro. The content-collections API maps directly to the command data model (§2) and handles the version/stage dimensions without custom code. Eleventy is a valid fallback if the team prefers minimal dependencies.

Bootstrap Commands (Astro, via Yarn 4)

yarn create astro@latest . -- --template minimal --no-install
yarn install
yarn dev          # local preview at http://localhost:4321
yarn build        # production output → dist/
CLAUDE.md constraint: All package operations must use yarn. Never run npm or npx. The yarn create invocation above satisfies this; Astro's own docs suggest npm create — ignore those instructions.

5. Content Distillation Pipeline

The RTF source is not directly machine-parseable by most SSG tooling. The distillation steps below transform it into the structured data and prose the site needs. 

flowchart TD
  A["YTbot Commands-VanillaVersion-EarlyGame.rtf\n(source of truth)"]
  A --> B["Manual extraction\nRead RTF, strip markup,\nidentify sections & rows"]
  B --> C["vanilla-early.json\nStructured command array\n(keyword, cost, effect, target, ...)"]
  B --> D["Prose sections → Markdown\nhow-it-works.md\nindex.md"]
  C --> E["Astro content collection\nsrc/content/commands/"]
  D --> E
  E --> F["yarn build\nAstro SSG"]
  F --> G["dist/ — static HTML/CSS\nready to deploy"]
  G --> H["Hosting\nGitHub Pages · Netlify · Vercel"]

RTF → structured data → SSG → deployment

Extraction Strategy

Since no RTF-to-text conversion tool (unrtf, pandoc, catdoc) is available in the environment, the extraction approach is:

  1. Read the RTF file in 150–200 line chunks (the file is ~963 lines).
  2. Identify text after control word sequences — prose appears after patterns like \loch\f45 and \hich\af45\dbch\af49\loch\f47.
  3. Author vanilla-early.json from the extracted command rows.
  4. Write prose sections in Markdown by paraphrasing extracted text (not verbatim copy to avoid RTF artifact noise).

6. Distillation Strategy: RTF → Scannable Docs

The RTF document is written as prose with embedded tables. The following treatments convert it into the scannable reference format viewers need mid-stream.

RTF Content Type Web Treatment Component / Pattern
Points rules (prose) Numbered list + callout box for inactivity rule Markdown prose in how-it-works.md
Patreon tier bonus table HTML table: Tier · Label · Point Multiplier Inline MDX table or BonusTable.astro
Super Chat bonus table HTML table: USD · Points Awarded Same pattern
Command syntax rules Code block with syntax diagram + targeting rules Fenced code + prose in commands.mdx
Command reference rows (~25+) Filterable table: Keyword · Category · Effect · Cost · Target CommandTable.astro — reads vanilla-early.json at build time
Spawn count formula Formula callout + table of spawn types with base/extra fields commands/spawns.mdx
Error / behavior rules Bulleted list with "gotcha" callout for no-refund rule Prose section at bottom of how-it-works.md

High-level implementation phases

Phase 1

Bootstrap SSG

  • Run yarn create astro@latest with minimal template
  • Confirm yarn dev serves a blank page
  • Add .editorconfig (2-space, LF, UTF-8) and .gitignore
  • Create src/content/commands/ directory and register the collection in astro.config.mjs
Phase 2

Extract & Structure RTF Content

  • Read RTF in chunks; manually extract all ~25+ command rows
  • Author src/content/commands/vanilla-early.json using the schema from §2
  • Author prose sections as Markdown files (index.md, how-it-works.md)
  • Cross-check all costs, effects, and notes against the RTF
Phase 3

Author Site Pages

  • Build CommandTable.astro component — reads vanilla-early.json, renders table with clickable column sort (vanilla JS, no framework)
  • Build commands.mdx — imports CommandTable, adds syntax section
  • Build how-it-works.md — points system, Super Chats, Patreon/Discord, behavior
  • Build commands/spawns.mdx — spawn mechanics, count formula, spawn-type table
  • Wire navigation in base layout
Phase 4

Style & Polish

  • Apply brand palette (dark theme matching stream aesthetic)
  • Add category color-coding to command rows (negative · positive · spawn)
  • Add @pseudoposse channel link in header/footer
  • Test responsive layout on mobile
Phase 5

Deploy

  • Finalize between GitHub Pages and Cloudflare Pages — both support Astro out of the box; Cloudflare gives preview deploys on PRs and a global CDN edge, GitHub Pages is simpler to configure for an existing GitHub org
  • Add yarn build → publish dist/ step in CI (GitHub Actions for either target)
  • Set site and base in astro.config.mjs to match the final URL
